# SRS – Система агрегатора служб транспортной доставки

## 1. Введение

### 1.1 Цель документа

Целью данного документа является описание требований к системе агрегатора служб транспортной доставки.

Система предназначена для объединения различных поставщиков услуг доставки в единую платформу, которая позволяет пользователям:

- создавать заказы на доставку
- сравнивать предложения различных служб доставки
- оплачивать доставку
- отслеживать статус выполнения заказа

Документ предназначен для следующих участников проекта:

- системных аналитиков
- разработчиков
- тестировщиков
- архитекторов системы
- менеджеров продукта

---

### 1.2 Область применения

Система агрегатора служб доставки предоставляет пользователям возможность оформления и управления заказами на доставку через единый интерфейс.

Основные возможности системы:

- создание заказа на доставку
- получение предложений от различных служб доставки
- выбор оптимального варианта доставки
- оплата доставки
- отслеживание статуса заказа

#### Не входит в область применения

- возврат заказов
- гарантийное обслуживание
- страхование доставки

---

### 1.3 Термины и определения

**Пользователь (User)**  
Зарегистрированный клиент системы, создающий заказ на доставку.

**Заказ (Order)**  
Набор данных, содержащий информацию о доставке, включая адрес, получателя и выбранную службу доставки.

**Агрегатор (Aggregator)**  
Сервис, объединяющий предложения нескольких служб доставки и предоставляющий пользователю оптимальные варианты.

**Событие (Event)**  
Сообщение, фиксирующее изменение состояния заказа.

## 2.2 Ограничения и допущения

### Технические ограничения

- пользователь должен быть авторизован в системе
- доступ к системе осуществляется через web-приложение
- интеграции реализуются через REST API
- обмен событиями осуществляется через брокер сообщений

### Бизнес-ограничения

- предложения служб доставки считаются актуальными
- система не отвечает за внутренние процессы логистических компаний
- обработка платежей осуществляется через внешний платежный шлюз

### Допущения

- пользователь вводит корректные данные
- службы доставки доступны через API
- платежные шлюзы возвращают ответ в установленное время

# 3. Бизнес-процессы

## 3.1 Основной процесс работы системы:

1. Пользователь создаёт заявку на доставку
2. Система отправляет запросы службам доставки
3. Получает предложения от поставщиков услуг
4. Пользователь выбирает оптимальный вариант
5. Пользователь выполняет оплату
6. Система создаёт заказ
7. Пользователь получает уведомление
8. Пользователь отслеживает статус доставки

# 5. Архитектура системы

## 5.1 Общая архитектура

Система построена по сервисной архитектуре и состоит из следующих компонентов.

### API Gateway

API Gateway является точкой входа для всех клиентских запросов.  
Основные функции компонента:

- маршрутизация запросов к соответствующим сервисам
- аутентификация и авторизация пользователей
- ограничение количества запросов (rate limiting)
- логирование и мониторинг входящих запросов

---

### Aggregator Service

Aggregator Service отвечает за агрегацию предложений различных служб доставки.

Основные функции сервиса:

- отправка запросов внешним службам доставки
- получение предложений от поставщиков доставки
- нормализация данных различных API
- формирование списка доступных вариантов доставки для пользователя

---

### Order Service

Order Service отвечает за управление заказами пользователей.

Основные функции сервиса:

- создание заказа
- хранение информации о заказе
- обновление статуса заказа
- предоставление информации о заказе по идентификатору
- публикация событий о заказах

---

### Payment Service

Payment Service отвечает за обработку платежей.

Основные функции сервиса:

- интеграция с внешними платежными шлюзами
- инициирование оплаты
- обработка результатов платежей
- обновление статуса заказа после оплаты

---

### Notification Service

Notification Service отвечает за уведомление пользователей.

Основные функции сервиса:

- отправка уведомлений о создании заказа
- уведомление об изменении статуса заказа
- отправка сообщений по email, SMS или push-уведомлениям

---

### Message Broker

Message Broker используется для асинхронного обмена событиями между сервисами.

Основные функции:

- публикация событий системы
- доставка событий подписчикам
- обеспечение слабой связанности между сервисами

---

### Database

База данных используется для хранения информации системы.

Основные данные:

- пользователи
- заказы
- платежи
- события системы


### Cache

Кэш используется для ускорения доступа к часто запрашиваемым данным и снижения нагрузки на основные сервисы и базу данных.


Основные сценарии использования кэша:

- хранение результатов агрегации предложений служб доставки
- кэширование часто запрашиваемых данных о заказах
- хранение пользовательских сессий
- ограничение количества запросов пользователей (rate limiting)
- временное хранение данных при обработке запросов

Основные данные:

- снизить нагрузку на базу данных
- уменьшить количество запросов к внешним API служб доставки
- ускорить время ответа системы


---

### 5.2 Модель данных

В системе используются следующие основные сущности данных.

### Users

Хранит информацию о пользователях системы.

Основные поля:

- user_id — уникальный идентификатор пользователя
- name — имя пользователя
- email — электронная почта
- phone — номер телефона
- created_at — дата регистрации пользователя

---

### Orders

Хранит информацию о заказах на доставку.

Основные поля:

- order_id — уникальный идентификатор заказа
- user_id — идентификатор пользователя
- delivery_provider_id — выбранная служба доставки
- pickup_address_id — адрес отправления
- delivery_address_id — адрес доставки
- delivery_price — стоимость доставки
- status — текущий статус заказа
- created_at — дата создания заказа

---


### OrderItems

Хранит информацию о содержимом отправления.

Основные поля:

- item_id — идентификатор объекта доставки
- order_id — идентификатор заказа
- item_name — название отправления
- weight — вес
- length — длина
- width — ширина
- height — высота

### Addresses

Хранит адреса отправления и доставки.

Основные поля:

- address_id — уникальный идентификатор адреса
- country — страна
- city — город
- street — улица
- building — номер дома
- apartment — квартира
- postal_code — индекс


### DeliveryProviders

Хранит информацию о подключённых службах доставки.

Основные поля:

- provider_id — идентификатор службы доставки
- provider_name — название службы доставки
- api_endpoint — URL API службы доставки
- status — статус активности интеграции

---

### Payments

Хранит информацию о платежах.

Основные поля:

- payment_id — уникальный идентификатор платежа
- order_id — идентификатор заказа
- payment_method — способ оплаты
- amount — сумма платежа
- payment_status — статус платежа
- created_at — дата создания платежа

---

### Events

Хранит события системы.

Основные поля:

- event_id — идентификатор события
- event_type — тип события
- entity_id — идентификатор связанной сущности
- payload — данные события, структура данных может отличаться в зависимости от типа события.
- created_at — дата создания события


---

# 6. Интеграции

## 6.1 API-системы

Основные API системы:
### POST /orders

Создание нового заказа.

**Назначение**

Позволяет пользователю создать новый заказ на доставку.

**Результат**

- создаётся новый заказ
- заказ получает уникальный идентификатор
- заказ сохраняется в системе

---

### GET /orders/{orderId}

Получение информации о заказе.

**Назначение**

Позволяет получить информацию о существующем заказе по его идентификатору.

**Результат**

- возвращается информация о заказе
- возвращается текущий статус заказа

---

### PATCH /orders/{orderId}

Обновление данных заказа.

**Назначение**

Позволяет обновлять состояние заказа (например статус оплаты или доставки).

**Результат**

- данные заказа обновляются
- возвращается обновлённая информация о заказе

## 6.2 Асинхронные интеграции

Система публикует события о заказах в брокер сообщений.  

### Топик
- **ORDERS.EVENTS** — основной топик для всех событий, связанных с заказами.

### Подписчики (Consumers)
Разные сервисы системы подписаны на топик и обрабатывают события в соответствии с задачами:  

- **Notification Service**  
  - Отправляет уведомления пользователю (email, SMS, push) о создании заказа, оплате и назначении доставки.  

- **Aggregator Service**  
  - Отслеживает изменения заказов для пересчёта доступных вариантов доставки.  

- **Order Service**  
  - Логирует ключевые события и обновляет внутреннее состояние заказа.
