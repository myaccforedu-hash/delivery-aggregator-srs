# SRS – Система агрегатора служб транспортной доставки

## 1. Введение

### 1.1 Цель документа

Целью данного документа является описание требований к системе агрегатора служб транспортной доставки.

Система предназначена для объединения различных поставщиков услуг доставки в единую платформу, которая позволяет пользователям:

- создавать заказы на доставку
- сравнивать предложения различных служб доставки
- оплачивать доставку
- отслеживать статус выполнения заказа

Документ предназначен для следующих участников проекта:

- системных аналитиков
- разработчиков
- тестировщиков
- архитекторов системы
- менеджеров продукта

---

### 1.2 Область применения

Функционал системы распространяется на следующие компоненты:

#### Frontend приложения

- Web-интерфейс пользователя
- Панель администратора

#### API Gateway

Точка входа для всех клиентских запросов.

#### Aggregator Service

Сервис агрегации предложений служб доставки.

#### Order Service

Основной сервис управления заказами.

#### Payment Service

Сервис интеграции с внешними платежными системами.

#### Notification Service

Сервис отправки уведомлений пользователям.

#### Message Broker (Kafka)

Используется для асинхронного обмена событиями между сервисами.

#### Базы данных

Используются для хранения:

- пользователей
- заказов
- платежей
- событий системы

##### Не входит в область применения

- возврат заказов
- гарантийное обслуживание товаров
- страхование доставки

---

### 1.3 Термины и определения

**Пользователь (User)**  
Зарегистрированный клиент системы, создающий заказ на доставку.

**Заказ (Order)**  
Набор данных, содержащий информацию о доставке, включая адрес, получателя и выбранную службу доставки.

**Агрегатор (Aggregator)**  
Сервис, объединяющий предложения нескольких служб доставки и предоставляющий пользователю оптимальные варианты.

**Событие (Event)**  
Сообщение, фиксирующее изменение состояния заказа.

- логистические системы

---

## 2.2 Ограничения и допущения

### Технические ограничения

- пользователь должен быть авторизован в системе
- доступ к системе осуществляется через web-приложение
- интеграции реализуются через REST API
- обмен событиями осуществляется через брокер сообщений Kafka

### Бизнес-ограничения

- предложения служб доставки считаются актуальными
- система не отвечает за внутренние процессы логистических компаний
- обработка платежей осуществляется через внешний платежный шлюз

### Допущения

- пользователь вводит корректные данные
- службы доставки доступны через API
- платежные шлюзы возвращают ответ в установленное время

---

# 3. Бизнес-процессы

## 3.1 BPMN процесс создания заказа

Основной процесс работы системы:

1. Пользователь создаёт заявку на доставку
2. Система отправляет запросы службам доставки
3. Получает предложения от поставщиков услуг
4. Пользователь выбирает оптимальный вариант
5. Пользователь выполняет оплату
6. Система создаёт заказ
7. Пользователь получает уведомление
8. Пользователь отслеживает статус доставки

# 4. Требования к системе

## 4.1 Функциональные требования

| Требование | Описание |
|-------------|-----------|
| FR-1 | Система должна позволять пользователю создавать заказ на доставку |
| FR-2 | Система должна получать предложения от нескольких служб доставки |
| FR-3 | Система должна отображать стоимость и сроки доставки |
| FR-4 | Система должна сохранять заказ в базе данных |
| FR-5 | Система должна интегрироваться с платежным шлюзом |
| FR-6 | Система должна публиковать события о заказах в Kafka |
| FR-7 | Система должна отправлять уведомления пользователю |
| FR-8 | Система должна поддерживать отслеживание доставки |
| FR-9 | Система должна обеспечивать отмену заказа |
| FR-10 | Система должна логировать ключевые события системы |

---

## 4.2 Нефункциональные требования

| Требование | Описание |
|-------------|-----------|
| NFR-1 | Система должна обеспечивать создание заказа и сохранение данных в базе не более чем за 3 секунды при стабильном интернет-соединении |
| NFR-2 | Система должна обеспечивать надежное хранение всех заказов в базе данных и предотвращать потерю данных при сбоях |
| NFR-3 | Система должна использовать защищённый протокол HTTPS для передачи данных между клиентом и сервером |
| NFR-4 | Система должна обеспечивать доступность сервиса не ниже 99.9% в месяц |
| NFR-5 | Система должна поддерживать горизонтальное масштабирование сервисов для обработки увеличивающегося количества заказов |
| NFR-6 | Пользовательский интерфейс должен обеспечивать возможность создания заказа не более чем за 3 шага |

# 5. Архитектура системы

## 5.1 Общая архитектура

Система состоит из следующих компонентов:

- API Gateway
- Aggregator Service
- Order Service
- Payment Service
- Notification Service
- Message Broker (Kafka)
- Database

---

## 5.2 Обоснование выбора базы данных

В системе используется **PostgreSQL**.

Причины выбора:

- высокая надежность
- поддержка транзакций
- хорошая масштабируемость
- широкая поддержка аналитических инструментов

---

## 5.3 Модель данных

Основные сущности системы:

- Users
- Orders
- DeliveryProviders
- Payments
- Events
