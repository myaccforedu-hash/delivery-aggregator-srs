# Order Service API (OpenAPI Specification)

## Описание

Данный папка репозитория содержит OpenAPI-спецификацию сервиса управления заказами транспортных услуг (Order Service).

Сервис отвечает за:

* создание заказа
* обновление параметров доставки
* управление статусами заказа
* отмену заказа
* интеграцию с асинхронными процессами (агрегатор доставки, платежи)

Спецификация разработана с учетом требований в задаче SA1347.

---

## Границы компонента

Компонент: Order Service

Ответственность:

* управление жизненным циклом заказа
* хранение и обновление данных
* публикация событий (вне API)

Внешние системы:

* Payment Service
* Aggregator Service
* Notification Service

---

## Основные возможности API

* Асинхронное создание заказа (POST /v1/orders)
* Получение заказа (GET /v1/orders/{id})
* Обновление заказа (PATCH /v1/orders/{id})
* Отмена заказа (DELETE /v1/orders/{id})
* Long-polling для статуса (GET /v1/orders/{id}/status)

---

## Архитектурные решения

### 1. HATEOAS (Level 3 REST)

API реализует уровень зрелости REST Level 3 за счет использования гипермедиа-ссылок.

Пример:

```json
{
  "orderId": "123",
  "status": "CREATED",
  "_links": {
    "self": { "href": "/v1/orders/123", "method": "GET" },
    "update": { "href": "/v1/orders/123", "method": "PATCH" },
    "cancel": { "href": "/v1/orders/123", "method": "DELETE" }
  }
}
```

Ссылки формируются динамически в зависимости от состояния заказа.

---

### 2. Асинхронность (Long Polling)

Создание заказа выполняется асинхронно:

* POST /v1/orders → 202 Accepted
* клиент использует:

  * Location
  * Retry-After

Polling:
GET /v1/orders/{id}/status

Возможные ответы:

* 200 OK — результат готов
* 204 No Content — обработка продолжается
* 408 Request Timeout — превышено время ожидания
* 424 Failed Dependency — ошибка внешнего сервиса

---

### 3. Идемпотентность

Для устранения повторного создания заказа используется:

Idempotency-Key (header)

Позволяет:

* безопасно повторять запросы
* избежать дублирования заказов

---

### 4. X-Correlation-Id (трассировка)

Используется заголовок:

X-Correlation-Id

Назначение:

* сквозная трассировка запроса через микросервисы
* логирование
* диагностика инцидентов

Передается:

* от клиента → в сервис
* от сервиса → в ответе

---

### 5. Обработка ошибок

Поддерживаются ключевые сценарии:

| Сценарий                   | Код |
| -------------------------- | --- |
| Ошибка валидации           | 400 |
| Не авторизован             | 401 |
| Конфликт состояния         | 409 |
| Ресурс заблокирован        | 423 |
| Превышение лимита запросов | 429 |
| Таймаут                    | 408 |
| Ошибка внешнего сервиса    | 424 |

---

### 6. Валидация данных

Используется строгая схема:

* additionalProperties: false — запрещены лишние поля
* ограничения:

  * maxLength
  * minItems
  * pattern
* enum для статусов
* обязательные поля (required)
* примеры (example)

---

### 7. JSON Schema

Все модели описаны через components/schemas и совместимы с JSON Schema Draft 4.
